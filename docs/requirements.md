# 需求文档：表格数据批量填充模板生成器

**版本**: v1.0 (MVP) | **日期**: 2026-05-14 | **状态**: 草稿

---

## 1. 概述

### 1.1 项目背景

设计师/运营人员需要将 Excel 中的一批商品、人物、卡片、海报等数据，逐条复制粘贴到 Figma 模板中，重复劳动量大且易出错。

### 1.2 产品定义

一款 Figma 插件，允许用户选择模板 Frame → 上传 Excel/CSV 表格 → 手动建立字段映射 → 按表格每一行批量生成模板副本并自动填入对应的文字和图片（图片为 Excel 单元格内嵌的实际图片对象）。

### 1.3 MVP 原则

- 窄场景闭环：只解决"表格 → 模板 → 批量生成"这一个核心链路
- 不做万能排版工具
- 不引入 AI 自动理解模板
- 不一开始支持所有 Excel 格式
- 先验证用户是否真的有高频痛点

---

## 2. 产品目标与用户价值

### 2.1 用户核心价值

| 价值 | 说明 |
|------|------|
| 减少重复劳动 | 避免对 50 行数据执行 50 次复制粘贴 |
| 减少操作错误 | 消除手工遗漏、错位、格式不一致 |
| 保持模板样式一致 | 生成结果与原模板字体、字号、颜色一致 |
| 提升批量设计效率 | 缩短从表格数据到设计成稿的交付时间 |

### 2.2 成功指标（待量化）

- 一次生成任务所花费的时间，相比人工操作降低的幅度
- 生成结果与模板的样式一致性（字体、字号、颜色保持不变）
- 用户完成一次批量为 N 行数据生成页面的操作步骤数

## 4. 用户故事

```
作为一个需要批量制作设计稿的人，
我已经设计好了一个模板，
我不想为 50 行数据重复复制粘贴 50 次，
我希望生成结果仍然像我手工排版的一样干净。
我想把表格中的每一行内容自动填入一个 Figma 模板，
以便快速生成一组视觉结构一致、内容不同的页面，
而不是手工重复复制文字、粘贴图片、调整位置和检查遗漏。
```

## 5. 领域模型

| 术语 | 解释 |
|------|------|
| **Source Table** （数据源表格） | 用户上传或粘贴的表格数据（Excel/CSV） |
| **Row**（数据行） | 表格中的一行，对应一个要生成的设计页 |
| **Field**（字段/列） | 表格中的一列，如标题、描述、价格、标签等文本数据，或 Excel 单元格中嵌入的图片 |
| **Template**（模板） | 用户在 Figma 画布中预先设计好的 Frame |
| **Placeholder**（占位层） | 模板中待被替换的文本层或图片层。模板内存在同名图层时，通过图层路径（如 "Frame > Group > Title"）区分 |
| **Mapping**（映射关系） | 用户指定的 "表格列 → 模板图层" 的对应关系 |
| **Generated Page**（生成页） | 根据某一行数据 + 模板生成的 Figma 设计内容 |
| **Batch**（批次任务） | 一次生成任务，包含多行数据及其生成结果 |
| **Issue**（问题行） | 某一行中已建立映射的文本字段，其对应单元格为空（字段未映射属于用户有意为之，不计为 Issue；图片提取失败不计为 Issue，仅作 warning） |

## 6. 功能性需求

### 6.1 数据源管理

- **FR-01**: 支持上传 Excel (.xlsx) 文件
- **FR-02**: 支持上传 CSV 文件
- **FR-03**: 插件解析表格后，在 UI 中展示字段（列名）清单。若文件无表头行，以 "列 A / 列 B / …" 作为默认列名
- **FR-04**: 展示数据行数预览。若数据行数为 0（仅表头或无有效数据），提示用户"未检测到数据行"并禁用生成按钮
- **FR-05**: 文件解析失败（损坏、不支持的格式、编码异常等）时，提示用户具体错误原因

### 6.2 模板选择

- **FR-06**: 用户在 Figma 画布中选择 **一个** Frame 作为模板。插件对选择的校验：
  - 未选中任何对象时，提示用户先选择一个模板 Frame
  - 选中多个对象时，提示用户仅选择一个 Frame
  - 选中对象非 Frame（如 Group、Component 实例等）时，提示用户选择一个 Frame
- **FR-07**: 插件识别并列出模板中可被映射的图层。扫描范围：模板 Frame 内的所有子孙文本层和图片层，但遇到嵌套 Frame（含 Component 实例）时停止深入，不进入其内部

### 6.3 字段映射

- **FR-08**: 用户手动建立 "表格列 → 模板图层" 的映射关系。同列不可重复映射到不同图层；不同列不可映射到同一图层
- **FR-09**: 支持文本列映射到文本层
- **FR-10**: 支持将 Excel **单元格中**嵌入的图片列映射到模板图片层（图片为 Excel 单元格内嵌的实际图片对象，非 URL 链接）
- **FR-11**: 允许部分字段不映射（未映射的图层保留模板原始内容）
- **FR-12**: 允许部分列不映射（未映射的列数据不参与生成）
- **FR-13**: 映射 UI 按列类型区分可映射目标：文本列仅可映射到文本层，图片列仅可映射到图片层；CSV 文件无图片列类型
- **FR-14**: 用户更改所选模板 Frame 时，清除已有映射并提示用户

### 6.4 批量生成

- **FR-15**: 点击"生成"后，插件遍历表格每一行数据。若未建立任何映射，仍允许生成（结果为一组模板副本，内容不做替换）, 但提示用户是否未选择映射时就开始生成.
- **FR-16**: 对每一行，克隆一份模板 Frame
- **FR-17**: 按映射关系，将对应列数据填入克隆 Frame 的对应图层
- **FR-18**: 生成的所有 Frame 排列在画布上，保持合理间距

### 6.5 预览与反馈

- **FR-19**: 生成前展示将生成的总页数
- **FR-20**: 生成过程中展示实时进度。支持用户取消操作；用户关闭插件面板时停止生成
- **FR-21**: 生成完成后展示结果摘要（成功 X 页，问题行 Y 行）。未建立任何映射时生成结果无"问题行"
- **FR-22**: 模板 Frame 不含任何文本层或图片层时，禁用生成按钮并提示"模板中无可填充的文本或图片图层"
- **FR-23**: Excel 单元格内嵌图片提取失败时，以 warning 方式提示用户，不标记为 issue，不阻塞该 Frame 的生成

### 6.6 容错处理

- **FR-24**: Excel 单元格内嵌图片提取失败时，对应图片位置不填充（保持空白），该 Frame 的其他字段正常填充
- **FR-25**: 已映射的文本字段对应单元格为空时，该文本层保留模板原始文本
- **FR-26**: 未映射的图层保留模板原始内容不变

---

## 7. 非功能性需求

### 7.1 样式一致性

- **NFR-01**: 生成页面中的文本需保持模板原有字体 (fontName)、字号 (fontSize)、颜色 (fills)、对齐方式 (textAlignHorizontal)
- **NFR-02**: 生成页面中的图片需保持模板原有尺寸和位置

### 7.2 性能

- **NFR-03**: 100 行以内的数据，生成总耗时不超过 30 秒
- **NFR-04**: 生成过程不阻塞 Figma UI（可取消操作）

### 7.3 兼容性

- **NFR-05**: 仅支持 Figma Editor 视图（不运行于 FigJam）
- **NFR-06**: 插件运行在 Figma 沙盒内，不发起外部网络请求

### 7.4 安全

- **NFR-07**: 数据文件在客户端本地解析，不上传至任何服务器

---

## 8. 用户工作流

> 步骤 1–4 顺序不强制，用户可先上传文件再选模板，或先开插件再选 Frame。插件应响应选择变化，而非要求特定操作顺序。

```
1. 用户在 Figma 画布中设计好一个模板 Frame
2. 用户选择该模板 Frame
3. 用户打开插件面板
4. 用户上传 Excel / CSV 文件
5. 插件展示字段列表和数据行数
6. 插件展示模板中可映射的占位层列表
7. 用户手动建立字段映射（如 "标题"列 → "Title"文本层）
8. 插件展示预览：将生成 X 页，Y 行数据存在潜在问题
9. 用户点击"生成"
10. 插件批量生成所有页面到画布
11. 用户检查、微调、交付
```

## 9. MVP 范围界定

### 9.1 本期包含

- Excel (.xlsx) 和 CSV 文件上传
- 单 Frame 模板选择
- 手动字段映射（文本列 → 文本层，Excel 内嵌图片列 → 图片层）
- 按行批量克隆模板并替换内容
- 基本容错（Excel 单元格内嵌图片提取失败仅 warning，不阻塞生成）

### 9.2 本期不包含

- AI 自动识别模板语义
- 多模板混合生成
- 图片 URL 链接读取（仅支持 Excel 内嵌图片）
- CSV 文件中嵌入图片（CSV 是纯文本格式，不支持图片）
- 复杂的布局自适应（如文本超长自动缩放/换行）
- 生成结果自动分组/自动创建 Page
- 模板嵌套 Frame 的深层递归匹配
- 协作/云端存储
- 撤销批量操作的回滚

## 10. 验收标准

### 验收样例 1：基础文本填充

```
Given 用户有一张包含 3 行数据的表格
  And 表格中有一列名为「标题」
  And 用户选择了一个包含标题占位文本层的模板
 When 用户把「标题」列映射到模板中的标题文本层
  And 点击生成
 Then 画布中生成 3 个页面（Frame）
  And 每个页面的标题分别来自对应行数据
  And 模板原有字体、字号、颜色保持不变
```

### 验收样例 2：文本 + 图片混合填充

```
Given 用户有一张包含 4 行数据的 Excel 表格
  And 表格中有一列名为「标题」，另有一列包含嵌入的图片
  And 用户选择了一个包含标题占位文本层和图片占位层的模板
 When 用户把「标题」列映射到标题文本层
  And 用户把图片列映射到图片层
  And 点击生成
 Then 画布中生成 4 个页面（Frame）
  And 每个页面的标题来自对应行，图片来自 Excel 对应行的嵌入图片
  And 模板原有字体、字号、颜色保持不变
  And 若某行单元格内嵌图片无法提取，该图片位置不填充（保持空白），不影响该行其他字段填充
```

### 验收样例 3：部分字段未映射

```
Given 用户上传了一个包含「标题」「描述」「价格」3 列的表格
  And 用户选择了一个包含标题、描述、价格、图片占位层的模板
 When 用户只将「标题」映射到标题文本层
  And 其他列不映射
  And 点击生成
 Then 生成的页面中，标题来自对应行数据
  And 其他图层（描述、价格、图片）保留模板原始内容
  And 未映射的列不影响生成结果
```

### 验收样例 4：文件上传与解析

```
Given 用户准备了一个包含表头「标题」「描述」和 10 行数据的 CSV 文件
 When 用户上传该文件
 Then 插件展示列名清单：标题、描述
  And 插件展示数据行数：共 10 行
```

### 验收样例 5：异常文件处理

```
Given 用户准备了一个已损坏的 xlsx 文件
 When 用户上传该文件
 Then 插件提示文件解析失败，并说明原因
  And 不进入映射和生成流程
```

### 验收样例 6：模板校验

```
Given 用户未选中任何对象
 When 用户在插件中查看模板选择区域
 Then 插件提示"请先选择一个模板 Frame"
  And 生成按钮不可用

Given 用户选中了一个 Group（非 Frame）
 When 插件检测到选择变化
 Then 插件提示"请选择一个 Frame，当前选中的是 Group"
```

### 验收样例 7：文本为空时的容错

```
Given 用户有一张 2 行数据的表格
  And 表格中「标题」列的第 2 行为空
  And 用户将「标题」列映射到模板标题文本层
 When 用户点击生成
 Then 第 1 页的标题来自第 1 行数据
  And 第 2 页的标题保留模板原始文本
  And 第 2 行被标记为问题行
```

### 验收样例 8：撤销已建立的映射

```
Given 用户已将「标题」列映射到模板文本层
  And 用户清除了该映射
 When 用户点击生成
 Then 生成的页面中，原映射目标层保留模板原始内容
```
